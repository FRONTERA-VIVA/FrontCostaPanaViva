1. VISIÓN GENERAL DEL PROYECTO
================================================================================

PROPÓSITO:
----------
Herramienta viva que conecta a los visitantes con los lugares, las personas y 
los negocios que hacen única a la región fronteriza entre Panamá y Costa Rica.

OBJETIVOS PRINCIPALES:
----------------------
1. Proporcionar información turística confiable y actualizada
2. Promover emprendimientos y negocios locales
3. Facilitar la exploración de la región fronteriza
4. Ofrecer información práctica para el tránsito fronterizo
5. Ser accesible sin necesidad de crear cuenta (panel público)

USUARIOS OBJETIVO:
------------------
1. Turistas nacionales y extranjeros
2. Emprendedores y comerciantes locales
3. Administradores de la plataforma

================================================================================
2. STACK TECNOLÓGICO
================================================================================

FRONTEND:
---------
- React 18 o superior (Componentes funcionales y hooks modernos)
- React Router (Navegación entre vistas - Doc 1, Req 19)
- Tailwind CSS (Diseño responsive y moderno)
- Leaflet.js + React Leaflet (Mapas interactivos - Doc 2, Sec 3)
- Axios (Consumo de API - Doc 1, Req 20)
- React Hook Form + Yup (Formularios y validación)
- i18next (Internacionalización ES/EN - Doc 3, Sec 5.7)

BACKEND:
--------
- Node.js + Express (API REST - Doc 2, Sec 2)
- PostgreSQL (Base de datos relacional - Doc 1, Sec 3.2)
- Prisma o Sequelize (ORM - Doc 2, Sec 2)
- JWT (Autenticación con tokens de acceso y refresco - Doc 2, Sec 3)
- Bcrypt (Encriptación de contraseñas - Doc 2, Sec 3)

SERVICIOS EXTERNOS:
-------------------
- Cloudinary (Almacenamiento y optimización de imágenes - Doc 2, Sec 2)
- Leaflet Tiles Provider (Mapas base)

DESPLIEGUE:
-----------
- Frontend: Vercel o Netlify (Doc 2, Sec 3)
- Backend: Railway o Render (Doc 2, Sec 3)
- Contenedores: Docker (opcional - Doc 2, Sec 3)

5. FLUJO DE DATOS
================================================================================

5.1 FLUJO PÚBLICO (Usuario sin cuenta)
--------------------------------------

1. Usuario abre la plataforma → Main.jsx monta App.jsx
2. App.jsx carga PublicLayout (Navbar + Footer + Contenido)
3. Usuario navega por Home/Explorer/Mapa/Details
4. Componentes solicitan datos vía services/ → api.js (Axios)
5. Backend responde con JSON → Componentes renderizan información
6. Usuario puede guardar favoritos (FavoritesContext - localStorage)
7. Usuario puede postular su negocio (Postulate.jsx → solicitudes.service.js)

5.2 FLUJO ADMINISTRADOR (Usuario autenticado)
---------------------------------------------

1. Admin accede a /admin/login → LoginForm.jsx
2. Credenciales validadas → auth.service.js → Backend
3. Backend retorna JWT (access + refresh tokens)
4. AuthContext guarda tokens y datos de usuario
5. Admin accede a rutas protegidas (ProtectedRoute wrapper)
6. AdminLayout carga con Sidebar + TopBar
7. Admin realiza CRUD → services/ → Backend → PostgreSQL
8. Logout invalida tokens y redirige a login

5.3 FLUJO DE IMÁGENES
---------------------

1. Usuario/Admin selecciona imagen → ImageUpload.jsx
2. Imagen se envía directamente a Cloudinary (preset configurado)
3. Cloudinary retorna URL de imagen optimizada
4. URL se guarda en PostgreSQL como referencia
5. Frontend carga imágenes desde Cloudinary con transformaciones
6. ServiceWorker cachea imágenes para modo offline (Doc 3, Sec 5.8.2)

5.4 FLUJO OFFLINE
-----------------

1. useOffline.js detecta pérdida de conexión
2. OfflineIndicator.jsx muestra banner de aviso
3. ServiceWorker sirve páginas cacheadas (Doc 3, Sec 5.8.1)
4. Formularios completados se guardan en IndexedDB
5. Al recuperar conexión, useOffline.js sincroniza datos pendientes
6. Usuario recibe confirmación de sincronización

6. INTEGRACIÓN CON BACKEND
================================================================================

6.1 ENDPOINTS PRINCIPALES (API REST)
------------------------------------

AUTENTICACIÓN:
  POST   /api/auth/login          → Iniciar sesión admin
  POST   /api/auth/refresh        → Refrescar token
  POST   /api/auth/logout         → Cerrar sesión

LUGARES TURÍSTICOS:
  GET    /api/lugares             → Listar lugares (público)
  GET    /api/lugares/:id         → Obtener detalle (público)
  POST   /api/lugares             → Crear lugar (admin)
  PUT    /api/lugares/:id         → Actualizar lugar (admin)
  DELETE /api/lugares/:id         → Eliminar lugar (admin)

EMPRENDIMIENTOS:
  GET    /api/emprendimientos     → Listar emprendimientos (público)
  GET    /api/emprendimientos/:id → Obtener detalle (público)
  POST   /api/emprendimientos     → Crear emprendimiento (admin)
  PUT    /api/emprendimientos/:id → Actualizar (admin)
  DELETE /api/emprendimientos/:id → Eliminar (admin)

EMPRESAS:
  GET    /api/empresas            → Listar empresas (público)
  GET    /api/empresas/:id        → Obtener detalle (público)
  POST   /api/empresas            → Crear empresa (admin)
  PUT    /api/empresas/:id        → Actualizar (admin)
  DELETE /api/empresas/:id        → Eliminar (admin)

SERVICIOS FRONTERIZOS:
  GET    /api/servicios-frontera  → Listar servicios (público)
  POST   /api/servicios-frontera  → Crear servicio (admin)
  PUT    /api/servicios-frontera/:id → Actualizar (admin)

CATEGORÍAS:
  GET    /api/categorias          → Listar categorías (público)
  POST   /api/categorias          → Crear categoría (admin)
  PUT    /api/categorias/:id      → Actualizar (admin)
  DELETE /api/categorias/:id      → Eliminar (admin)

SOLICITUDES:
  POST   /api/solicitudes         → Enviar postulación (público)
  GET    /api/solicitudes         → Listar solicitudes (admin)
  PUT    /api/solicitudes/:id     → Revisar solicitud (admin)

ANALÍTICA:
  GET    /api/analytics/estadisticas  → Estadísticas dashboard (admin)
  GET    /api/analytics/busquedas     → Búsquedas frecuentes (admin)
  GET    /api/analytics/vistas        → Registros más vistos (admin)

6.2 CONFIGURACIÓN DE AXIOS (src/config/api.js)
----------------------------------------------

- Base URL desde variables de entorno (.env)
- Interceptor de request: Adjunta token JWT automáticamente
- Interceptor de response: Maneja errores 401 (refresh token)
- Timeout configurado para redes móviles
- Headers comunes (Content-Type: application/json)

6.3 MANEJO DE ERRORES
---------------------

- Errores 400: Validación de datos (mostrar mensajes de Yup)
- Errores 401: Redirigir a login o refrescar token
- Errores 403: Mostrar mensaje de permiso denegado
- Errores 404: Mostrar página no encontrada
- Errores 500: Mostrar mensaje genérico de error del servidor
- Errores de red: Activar OfflineIndicator y reintentar

9. CONSIDERACIONES DE SEGURIDAD
================================================================================

9.1 AUTENTICACIÓN Y AUTORIZACIÓN
--------------------------------

- JWT con tokens de acceso (15 min) y refresco (7 días)
- Tokens almacenados en httpOnly cookies (producción)
- Refresh token rotativo con invalidación de antiguos
- Roles de usuario: admin, moderador, visualizador
- ProtectedRoute wrapper para rutas privadas
- Interceptors de Axios para manejo automático de tokens

9.2 VALIDACIÓN DE DATOS
-----------------------

- Validación en cliente con Yup (formularios)
- Validación en servidor (backend - obligatorio)
- Sanitización de inputs para prevenir XSS
- Prepared statements vía ORM para prevenir SQL Injection
- Límites de tamaño en uploads de imágenes

9.3 COMUNICACIÓN SEGURA
-----------------------

- HTTPS obligatorio en producción
- Headers de seguridad (CORS, CSP, HSTS)
- Rate limiting en endpoints críticos (login, postulación)
- Logs de auditoría para acciones administrativas

9.4 PROTECCIÓN DE RECURSOS
--------------------------

- Imágenes servidas desde Cloudinary (no desde servidor)
- Assets estáticos con cache control
- API endpoints protegidos por autenticación cuando corresponde
- Información sensible nunca expuesta en respuestas de API
